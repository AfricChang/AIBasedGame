# 移动端 2D 迷宫游戏 ExecPlan

更新时间：2026-06-28

说明：

- 仓库中未发现 `agent/PLANS.md` 或 `agent/PLANS.2.md` 文件。
- 本文档按 ExecPlan 思路给出设计评审后的实施路径，供批准后直接执行。

## 1. 目标

在评审通过后，分阶段交付一个可在移动端浏览器和现有 Cordova 壳内运行的 2D 迷宫游戏，满足：

- 20 个关卡
- 渐进式视野
- 计时和结果结算
- 响应式移动端 UI
- Buff 扩展接口
- 可嵌入当前项目，也可单独打包

## 2. 分阶段计划

### 阶段 A：项目骨架与基础运行时

产出：

- 基础目录结构
- `index.html`
- 全局样式
- 主菜单、设置、关于、关卡选择骨架
- `GameApp`、`SceneRouter`、`StorageService`

验收标准：

- 页面可在手机浏览器正常打开
- 菜单流转完整
- 设置可持久化

### 阶段 B：核心迷宫玩法

产出：

- 网格地图模型
- 角色四向移动
- 地图渲染
- 计时器
- 基础通关结算

验收标准：

- 玩家可完成至少 3 个基础关
- 计时、暂停、重开可用

### 阶段 C：视野系统

产出：

- `VisibilitySystem`
- `hidden / explored / visible` 三态
- 半径配置
- 暗区与可见区渲染

验收标准：

- 玩家移动时视野实时更新
- 墙后区域不会误显示
- 调整半径不需要改业务代码

### 阶段 D：关卡机制与 Buff 接口

产出：

- 钥匙门
- 信标
- 减速地块
- 传送门
- 限时规则
- `BuffSystem`

验收标准：

- 至少 8 个机制关可顺利运行
- 临时全图视野和速度提升 Buff 可工作

### 阶段 E：20 关落地与调优

产出：

- 20 个正式关卡
- 解锁进度
- 最佳成绩记录
- 失败界面与通关界面

验收标准：

- 20 关全部可达且可通关
- 难度曲线可接受

### 阶段 F：移动端打磨与接入

产出：

- 输入灵敏度优化
- 适配主流手机尺寸
- 资源压缩
- 接入现有游戏集合入口
- 输出嵌入版入口
- 输出独立版入口
- 同步到 `game-app/www/`

验收标准：

- Android WebView 可运行
- 主要页面无布局溢出
- 操作延迟可接受
- 嵌入版与独立版均可正常启动、暂停和退出

### 阶段 G：轻量表现增强

产出：

- 程序化木质地板
- 更清晰的墙体、机关和特殊格视觉层级
- 基于 `Web Audio API` 的短音效反馈
- 设置页音效开关
- 可选外部素材包调研记录

验收标准：

- 地板纹理可提升质感，但不干扰迷宫墙体、单向门、钥匙、出口等关键信息
- 连续移动时音效不刺耳、不堆叠失控
- 音效开关可持久化
- 不引入未确认授权的第三方素材
- 不明显增加独立版包体

## 3. 开发顺序建议

推荐顺序：

1. 菜单和场景骨架
2. 单关最小可玩原型
3. 视野系统
4. 关卡机制
5. Buff 系统
6. 20 关内容化
7. 移动端优化
8. 程序化地板与轻量音效

## 4. 测试计划

### 自动测试

- 迷宫生成一致性
- BFS 可达性校验
- 视野算法单元测试
- Buff 生命周期单元测试
- 关卡数据 schema 校验

### 手动测试

- iPhone 尺寸浏览器
- 常见安卓尺寸浏览器
- 竖屏优先场景
- 暂停/恢复/后台切换
- 低性能设备连续操作

## 5. 风险控制

1. 若视野算法性能不达标，优先优化为“移动后再重算”，而不是每帧重算。
2. 若后期机关过多导致手感恶化，优先削减机制数量，不牺牲基础可玩性。
3. 若双目录同步成本高，先只在 `MobileMazeGame/` 迭代，通过后再导出。
4. 若独立壳与嵌入壳出现行为差异，优先把差异收敛到 `HostAdapter`，不在业务层分叉。
5. 若外部视觉或音频素材授权不清，优先使用程序化 Canvas 绘制和 Web Audio 参数音效。
6. 若木质纹理影响可读性，降低纹理对比度或只保留可见区弱纹理。

## 6. 评审通过后的首批交付建议

建议第一轮编码只交付：

- 菜单骨架
- 3 个基础关
- 视野系统
- 计时
- 关卡结算

原因：

- 能最快验证核心玩法是否成立
- 能在低风险下确认触屏交互是否舒服
- 能在完整铺开 20 关之前及早修正方向
